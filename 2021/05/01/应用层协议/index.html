这篇博客我们聊一聊应用层的常见应用，以及其所采用的协议。

应用程序结构

应用程序结构（Application architecture）包括 Client-server 和 Peer-to-peer（P2P）两种形式。

Client-server 结构

服务端（server）具有如下特点：

  • 永远处于启动状态
  • 具有长久不变的 IP 地址
  • 常由组织构成服务器农场（server farm）提供强大的服务器功能

客户端（client）具有如下特点：

  • 与服务端通信
  • 可能间断性地连接
  • 会有动态（改变的）IP 地址
  • 不直接与另一客户端通信

P2P 结构

P2P 结构的特点是：

  • 没有永远运行着的服务器
  • 任意的端系统可直接通信
  • Peers 间断性连接并改变 IP 地址

P2P 可扩展性很强，但是也更难管理。

Client & P2P 混合结构

现实中也会使用 Client-server 和 P2P 的混合结构。例如 Skype 和 Instant messaging，两个用户聊天是 P2P 实现的，而用户的登录/定位是通过 central server 实现的。

进程通信

进程是在主机内部运行的程序。在同一主机内，两个进程使用进程间通信（inter-process communication），由 OS 控制。在不同主机之间，进程通过交换信息（exchanging messages）实现通信。

进程分为客户端进程和服务端进程，由客户端进程初始化通信，服务端进程等待被连接。P2P 引用程序将同时包含客户端进程和服务端进程。

进程寻址 Addressing processes

为了准确地收到消息，进程必须要有标识符（identifier）。首先，每个主机（host）都有特有的 32 位 IP 地址，但是这样无法区别运行在同一主机上不同进程。因此，进程的标识符包括 IP 地址和端口号（port number）。例如，HTTP 服务端口号为 80，邮件服务端口号为 25。

互联网传输协议服务 Transport Protocol

传输协议主要包括 TCP 服务和 UDP 服务。

TCP 服务

TCP 服务具有如下特点：

  • 需要在 client 和 server 之间建立连接
  • 能保证可靠的传输
  • 流控制 flow control：控制 sender 的发送不会超量
  • 拥塞控制 congestion control：当网络超载时，阻塞 sender
  • 不提供计时，最小带宽保证的功能

UDP 服务

UDP 服务具有如下特点：

  • 提供的数据传输是不可靠的
  • 不提供：建立连接、可靠性、流控制、拥塞控制、计时或带宽保证

Web 和 HTTP

基本术语

Web page 是由 objects 组成的，objects 包括：HTML 文件、JPEG 图片、Java 程序、音频文件。

Web page 由基本的 HTML 文件构成，在文件中包含了一些 URL 来引用所需的 objects，例如：

www.someschool.edu/somDept/pic.gif

HTTP 协议

超文本传输协议（hypertext transfer protocol，HTTP）是 Web 应用层协议，采用 client-server 模型，其中：

  • client: 浏览器，用于请求、接收、展示 Web objects
  • server: Web 服务器，根据来自 client 的请求，发送对应 objects

HTTP 协议所采用的传输协议是 TCP 协议，HTTP 服务器对于 client requests 的历史记录是没有保存的（stateless）。整个交互流程如下：

  1. client 初始化 TCP 连接：创建连接到 server 的 socket，端口号为 80
  2. client

HTTP 协议包括 Non-persistent HTTP 和 Persistent HTTP 两种。

Non-persistent HTTP

  • 每个 TCP 连接最多只能发送一个 object
  • HTTP/1.0 使用的是 non-persistent HTTP
  • 每个 object 需要 2 RTTs 的时间
  • 每个 TCP 连接都会导致 OS overhead
  • 浏览器通常使用并行的 TCP 连接来获取 referenced objects

Persistent HTTP

  • 一个 TCP 连接可以发送多个 objects

  • HTTP/1.1 默认使用 persistent connections

  • server 保持连接开启，之后的 HTTP 消息都通过这个TCP连接传输

  • without pipelining: client 再收到上一个回应，才会进行新的请求，因此每个获取 object 需要消耗一个 RTT 的时间

  • with pipelining: client 每遇到一个 referenced object 就发送一个请求，HTTP/1.1 默认

Round Trip Time

Round Trip Time（RTT）是一个 packet 从 client 传输到 server 再传输回来所用的时间。在HTTP交互中，需要 2RTT​ 的时间，1RTT 来初始化 TCP 连接，1RTT​ 来发送 HTTP 请求和接收 HTTP 回应。总的文件传输时间为： \[ total=2RTT+<\mathrm{file\ transmit\ time}> \]

HTTP response status codes

HTTP server 在回复 client 时会带有一个状态码（status code），下表列出了常见的状态码。

status code Description
200 OK request succeeded, requested object later in this message
301 Moved Permanently requested object moved, new location specified later in this message
400 Bad Request request message not understood by server
404 Not Found request document not found on this server
505 HTTP Version Not Supported server doesn't support the HTTP version of the client

Cookies

Cookies 的作用是保存 user-server 的状态，常见于很多网站。

Web caches (proxy server)

Web 缓存，又称代理服务器，的目标是能够在不通过 server 的情况下，满足 client request。它的好处是：

  • 可以减少 client request 的响应时间
  • 可以减小链路上的阻塞问题
  • 可以让差的内容提供者更有效地提供内容

Conditional GET

那么 HTTP 也要做出相应改变来兼容 Cache 的功能，它的方法就是使用 Conditional GET。在 Conditional GET 中，加入了 If-modified-since <date> 字段，发送给 server。该字段中表示的是 cache 中保存的该 object 的最新修改的时间。如果 server 发现自 date 之后，这个 object 没有进行过更新，那么就返回 304 Not Modified，表示 client 可以从 cache 直接获取该 object。如果该 object 在 date 之后更新过了，那么 server 需要返回更新的 object 给 client。

FTP

简介

文件传输协议（the File Transfer Protocol，FTP）处理的是对远程主机收发文件的服务，所使用的依然是 client-server 模型，其中：

  • client: 初始化传输任务
  • server: 远程主机

流程

FTP 信息传输使用的也是 TCP 协议，整个 FTP 的协议流程如下：

  • FTP client 通过端口 21 联系 FTP server，获取 server 的许可
  • Client 通过发送 control commands 来浏览远程文件夹
  • 当 server 收到 command 时，开启通向 client 的 TCP 数据连接
  • 在传输一个文件之后，server 关闭连接
  • Server 开启另一个 TCP 数据连接来传输另一个文件
  • FTP 的 control connection 是 out of band 的，即：数据消息和控制消息使用两条不同的连接实现。在 FTP 中，control connection 在 port 21 上，data connection 在 port 20 上（不绝对）。
  • FTP server 维护状态信息，包括：当前文件夹、之前的认证信息...

  • User agents：主要用于创作、编辑、阅读邮件信息
